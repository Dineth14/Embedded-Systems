Modeling

UML for Embedded Systems

UML provides standardised diagrams to capture structure and behaviour before writing code. The most useful diagrams for embedded work:

DiagramTypeCaptures
ClassStructuralModule interfaces, data types, relationships
SequenceBehaviouralMessage flow between objects over time
State MachineBehaviouralEvent-driven behaviour with states & transitions
ActivityBehaviouralControl flow — similar to flowcharts
Why Model?

Models help catch design errors early — when fixing them costs minutes, not weeks. A state machine diagram reviewed before coding can eliminate entire classes of bugs.

Finite State Machines

Definition

A Mealy FSM is a tuple \((S, \Sigma, \Lambda, \delta, \omega, s_0)\) where \(S\) = states, \(\Sigma\) = input alphabet, \(\Lambda\) = output alphabet, \(\delta: S \times \Sigma \rightarrow S\) = transition function, \(\omega: S \times \Sigma \rightarrow \Lambda\) = output function, \(s_0\) = initial state. A Moore FSM has \(\omega: S \rightarrow \Lambda\) (output depends only on state).

FSM Implementation Patterns

Switch-Case

typedef enum { ST_RED, ST_GREEN, ST_YELLOW } state_t;
state_t state = ST_RED;

void fsm_step(event_t evt) {
    switch (state) {
    case ST_RED:
        if (evt == EVT_TIMER) {
            set_light(GREEN);
            start_timer(30000);
            state = ST_GREEN;
        }
        break;
    case ST_GREEN:
        if (evt == EVT_TIMER) {
            set_light(YELLOW);
            start_timer(5000);
            state = ST_YELLOW;
        }
        break;
    case ST_YELLOW:
        if (evt == EVT_TIMER) {
            set_light(RED);
            start_timer(45000);
            state = ST_RED;
        }
        break;
    }
}

Table-Driven

typedef struct {
    state_t  current;
    event_t  event;
    state_t  next;
    void   (*action)(void);
} transition_t;

const transition_t table[] = {
    { ST_RED,    EVT_TIMER, ST_GREEN,  action_green  },
    { ST_GREEN,  EVT_TIMER, ST_YELLOW, action_yellow },
    { ST_YELLOW, EVT_TIMER, ST_RED,    action_red    },
};

void fsm_dispatch(event_t evt) {
    for (int i = 0; i < ARRAY_SIZE(table); i++) {
        if (table[i].current == state &&
            table[i].event   == evt) {
            table[i].action();
            state = table[i].next;
            return;
        }
    }
}
Trade-offs

Switch-case: Simple, fast, easy to debug; grows unwieldy beyond ~10 states. Table-driven: Scales well, data-driven, easier to auto-generate; slightly more RAM, harder to step through in debugger.

Hierarchical State Machines (HSM)

HSMs (Harel statecharts) add hierarchy: a superstate contains substates. Common behaviour is handled at the parent level, reducing duplication.

Motor Controller HSM

Superstate: OPERATIONAL (handles FAULT → ERROR globally)

  • Sub: IDLE — waiting for START command → RUNNING
  • Sub: RUNNING — speed control loop. STOP → BRAKING
  • Sub: BRAKING — ramp-down. speed=0 → IDLE

Any FAULT event in any substate propagates to parent OPERATIONAL, which transitions to ERROR — no need to duplicate the handler in each substate.

Event Delegation

state_t Running_handler(event_t evt) {
    switch (evt) {
    case EVT_STOP:  return ST_BRAKING;
    default:        return Operational_handler(evt); // delegate to parent
    }
}

Model-Driven Development

Tools like Simulink Stateflow, YAKINDU, and Quantum Leaps QP generate C code directly from state machine models, ensuring the implementation matches the design.

Benefits
  • Visual design that non-programmers (systems engineers) can review
  • Automatic code generation eliminates hand-coding errors
  • Formal verification possible on the model before deployment
  • Traceability from requirements → model → generated code
