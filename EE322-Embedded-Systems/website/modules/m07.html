Task Scheduling

Scheduling Overview

Real-time scheduling assigns CPU time to tasks such that all timing constraints are met. A schedule is feasible iff every task meets every deadline.

PropertyDescription
\(T_i\)Period of task \(i\)
\(C_i\)Worst-case execution time (WCET)
\(D_i\)Relative deadline (often \(D_i = T_i\))
\(U_i = C_i / T_i\)Utilisation of task \(i\)

Total utilisation: \(U = \sum_{i=1}^{n} \frac{C_i}{T_i}\). Necessary condition: \(U \leq 1\).

Rate-Monotonic Scheduling (RMS)

RMS Rule

Assign highest priority to the task with the shortest period. RMS is the optimal static-priority algorithm for periodic, preemptive, independent tasks with \(D = T\).

Liu & Layland utilisation bound:

$$U_{\text{lub}} = n \left( 2^{1/n} - 1 \right)$$

Key Insight

If \(U \leq U_{\text{lub}}\), all deadlines are guaranteed under RMS. If \(U_{\text{lub}} < U \leq 1\), you must verify with Response-Time Analysis.

Earliest Deadline First (EDF)

EDF Rule

At every scheduling decision, execute the ready task with the closest absolute deadline. EDF is the optimal dynamic-priority algorithm for preemptive uniprocessor systems.

Schedulability test: the task set is feasible under EDF iff:

$$U = \sum_{i=1}^{n} \frac{C_i}{T_i} \leq 1$$

Schedulability Analysis

Response-Time Analysis (RTA)

For fixed-priority scheduling, compute worst-case response time \(R_i\) iteratively:

$$R_i^{(k+1)} = C_i + \sum_{j \in hp(i)} \left\lceil \frac{R_i^{(k)}}{T_j} \right\rceil C_j$$

Start with \(R_i^{(0)} = C_i\) and iterate until convergence. Task \(i\) is schedulable iff \(R_i \leq D_i\).

Worked Example

Three tasks: \(\tau_1(T{=}10, C{=}3)\), \(\tau_2(T{=}15, C{=}4)\), \(\tau_3(T{=}20, C{=}5)\).

\(U = 0.3 + 0.267 + 0.25 = 0.817\). \(U_{\text{lub}}(3) = 0.780\) → exceeds bound, use RTA.

\(R_3^{(0)} = 5\), \(R_3^{(1)} = 5 + \lceil5/10\rceil\cdot3 + \lceil5/15\rceil\cdot4 = 5+3+4 = 12\), \(R_3^{(2)} = 5 + \lceil12/10\rceil\cdot3 + \lceil12/15\rceil\cdot4 = 5+6+4 = 15\), \(R_3^{(3)} = 5+6+4 = 15\) → converged. \(R_3 = 15 \leq 20\) ✓.

WCET & Measurement

Worst-Case Execution Time must be bounded for hard real-time tasks:

  • Static analysis: Abstract interpretation (e.g., aiT) — safe bounds, pessimistic
  • Measurement-based: Hardware timers / DWT cycle counter — practical but finds observed WCET, not true WCET
  • Hybrid: Combine both approaches
// Measure with DWT cycle counter (Cortex-M)
CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
DWT->CYCCNT = 0;
DWT->CTRL   |= DWT_CTRL_CYCCNTENA_Msk;

uint32_t start = DWT->CYCCNT;
my_critical_function();
uint32_t elapsed = DWT->CYCCNT - start;
